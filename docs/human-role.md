# 人在 OAR 中的作用

更新日期：2026-05-26

## 1. 核心判断

人在 OAR 中不是被智能体替代的执行者，而是 **目标判断、授权和组织承诺的负责人**。

OAR 的产品边界应该是：

> 智能体负责观察、诊断、起草和准备动作；人负责判断、授权、校准和承担组织责任。

一句话：

> 人是目标系统里的驾驶员和责任人，智能体是副驾、参谋和运营执行器。

这个边界是 OAR 能被企业接受的关键。如果产品变成“全自动管理 OKR”，风险会很高；如果产品是“帮人把目标运营工作准备好、排好队、可确认地执行”，机会更大。

## 2. 人与智能体的分工

| 维度 | 人负责 | 智能体负责 |
| --- | --- | --- |
| 目标意义 | 判断目标是否还重要、是否要调整 | 发现目标是否失焦、是否缺进展 |
| 风险判断 | 决定风险是否真实、优先级多高 | 找出长期未更新的 KR、低进度、缺证据、阻塞点 |
| 组织动作 | 决定是否提醒、约会、写评论、更新进度 | 起草提醒、会议、评论、复盘、建议动作 |
| 权限授权 | 确认哪些动作可以写回飞书 | 生成 dry-run、执行请求和审计日志 |
| 信任校准 | 反馈“这个建议为什么不对” | 记住被拒绝原因，下次减少同类建议 |
| 责任归属 | 对最终写回和组织影响负责 | 提供可追溯、可解释、可审计的过程 |

核心边界：

> 智能体可以自动判断和起草，但不能自动代表人做组织承诺。

## 3. 人的五个关键作用

### 3.1 定目标

智能体可以发现目标偏离、目标缺证据、KR 长期不更新，但不能替团队决定公司到底要不要继续押这个方向。

目标是否仍然重要、是否需要调整、是否要降优先级，必须由人判断。

### 3.2 判轻重

同样是 KR 落后，可能代表战略风险，也可能只是目标已经不重要，或者数据还没有回填。

智能体能提供证据和推断，但风险优先级、组织语境和业务取舍要由人决定。

### 3.3 授权动作

写评论、提醒 owner、创建任务、更新进度、约会议，都会影响组织关系和协作节奏。

因此 OAR 的写回动作必须进入待确认队列，由人确认、编辑后确认或拒绝。

### 3.4 校准智能体

人的确认、编辑和拒绝，不只是交互结果，也是系统学习信号。

OAR 应该记录：

- 哪类建议被确认。
- 哪类建议被编辑。
- 哪类建议被拒绝。
- 用户拒绝的原因。
- 哪些证据被认为可信或不可信。

这些反馈应进入决策记忆和策略规则，让智能体逐步贴近团队真实的运营方式。

### 3.5 承担责任

系统可以提供证据链、审计日志和执行记录，但组织承诺不能甩给模型。

最终写回飞书、通知成员、创建任务、推动协作的责任主体仍然是人。

## 4. 产品设计原则

OAR 的交互不应该让用户感觉“智能体已经替你决定了”，而应该让用户感觉：

> 智能体已经把信息查好、风险排好、动作写好，现在等你做判断。

因此产品设计应遵循：

- 首页是复盘收件箱，不是聊天框。
- 智能体建议必须有证据链。
- 所有写回动作必须有确认入口。
- `Confirm` 之前必须能看到目标、内容、身份、影响范围和 dry-run 结果。
- `Reject` 必须允许记录原因。
- `Edit then Confirm` 应该是一等能力，不是边缘功能。
- 审计时间线必须能解释“谁基于什么证据授权了什么动作”。

## 5. 权限边界

人在 OAR 中不仅是用户，也是授权主体。

权限边界应该是：

- 智能体只能读取当前用户或企业授权范围内的数据。
- 智能体不能自动扩大 scope。
- 智能体不能绕过 OAR policy allowlist。
- 智能体不能直接持有长期 token。
- 智能体不能直接对外部 A2A agent 暴露原始飞书数据。
- 写回必须来自人的 `ConfirmedAction`。

这意味着 Lark 登录只是身份绑定，不等于智能体拥有用户全部权限。

实际可用权限必须取以下交集：

> 飞书 app scopes + 用户授权 + 资源权限 + OAR 策略 allowlist + 人的确认。

## 6. 对 MVP 的影响

MVP 里人的核心动作应该非常清楚：

1. 打开每周复盘收件箱。
2. 查看本周 OKR 风险。
3. 查看每条风险背后的证据链。
4. 对建议动作选择确认、编辑后确认或拒绝。
5. 在审计时间线中看到动作结果。

MVP 不应该追求“智能体全自动完成 OKR 管理”，而应该追求：

> 让 manager / PMO 每周更快做出目标执行判断，并更安全地推动行动闭环。

## 7. 长期形态

当 OAR 从 OKR 复盘驾驶舱演进为目标执行智能运营系统时，人的角色不会消失，而会升级：

- 从手动收集信息，变成判断智能体整理出的风险。
- 从手动写周报，变成确认和校准每周简报。
- 从手动催进展，变成授权系统发起跟进行动。
- 从被动追问题，变成主动运营目标执行风险。

最终形态不是“无人化管理”，而是：

> 人做更高质量的判断，智能体承担更重的运营准备和执行编排。
