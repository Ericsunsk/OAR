# OAR 记忆架构

更新日期：2026-05-26

## 1. 核心结论

OAR 的记忆不建议只做两层。推荐采用 **三层主架构**：

> 结构记忆 / Ontology Graph + 语义记忆 / Vector Memory + 决策记忆 / Decision Memory。

这三层分别回答不同问题：

| 层 | 回答的问题 | 作用 |
| --- | --- | --- |
| 结构记忆 / Ontology Graph | 谁和谁有关？什么对象和什么对象有关？ | 梳理组织、目标、任务、文档、会议、动作之间的稳定关系 |
| 语义记忆 / Vector Memory | 哪些内容语义相关？哪里提到过类似问题？ | 从文档、会议纪要、任务评论、历史复盘中找上下文 |
| 决策记忆 / Decision Memory | 人过去怎么判断？哪些建议被接受或拒绝？ | 记录人的确认、编辑、拒绝和偏好，让系统越用越贴合团队 |

一句话：

> Ontology Graph 是骨架，Vector Memory 是语义检索，Decision Memory 是组织判断的学习层。

如果只做“图谱 + 向量”两层，OAR 会像一个聪明搜索系统；加上决策记忆后，OAR 才会逐渐像一个懂团队目标运营方式的智能运营系统。

## 2. 为什么需要三层

### 2.1 只有图谱不够

图谱能告诉系统：

- 这个 KR 属于哪个 Objective。
- owner 是谁。
- 关联了哪些任务、文档、会议。
- 哪些动作曾经写回过飞书。

但图谱不擅长回答：

- 最近会议里真实的阻塞点是什么。
- 某个延期原因在文档中是如何被描述的。
- 用户为什么拒绝了上次提醒 owner 的建议。

### 2.2 只有向量不够

向量检索能找相似语义内容，但它不天然理解：

- 组织层级。
- owner 关系。
- OKR 对齐关系。
- 权限边界。
- 动作是否已经被确认或拒绝。

如果只有向量，系统容易变成“相关内容搜索”，而不是“目标执行运营”。

### 2.3 缺少决策记忆会失去产品壁垒

OAR 最有价值的记忆不是“飞书里有什么”，而是：

> 这个组织如何判断目标风险，如何授权动作，以及什么建议对他们可信。

例如：

- 某个 manager 不喜欢系统直接催 owner，更喜欢先在周会里同步。
- 某个团队接受简短评论，不接受长篇 AI 总结。
- 某类 blocker 过去被误判过，下次应降低置信度。
- 某个证据来源经常滞后，不应该作为强证据。

这些都属于决策记忆，是 OAR 越用越聪明的关键。

## 3. 三层架构

### 3.1 结构记忆 / Ontology Graph

结构记忆用于表达稳定实体和关系。

第一版核心实体：

| 实体 | 说明 |
| --- | --- |
| `Tenant` | 飞书企业租户，数据隔离边界 |
| `User` | 飞书用户 / OAR 用户 |
| `Team` | 团队或组织单元 |
| `OkrCycle` | OKR 周期 |
| `Objective` | 目标 |
| `KeyResult` | 关键结果 |
| `Task` | 飞书任务 |
| `Doc` | 飞书文档 / Wiki |
| `Meeting` | 会议 / 日历事件 / 会议纪要 |
| `Evidence` | 证据摘要和来源引用 |
| `ProposedAction` | 智能体建议动作 |
| `ConfirmedAction` | 人确认后的动作 |
| `DecisionFeedback` | 用户确认、编辑、拒绝及原因 |

第一版核心关系：

| 关系 | 含义 |
| --- | --- |
| `belongs_to` | 对象属于某个租户、团队、周期或目标 |
| `owns` | 用户拥有 Objective / KR / Task |
| `aligns_with` | Objective / KR 之间存在对齐关系 |
| `mentions` | 文档、会议、任务评论提到了某个 OKR 对象 |
| `supports` | 证据支持某个判断或建议 |
| `blocks` | 某个证据或任务阻塞某个 KR |
| `derived_from` | 建议动作来源于某些证据 |
| `confirmed_by` | 动作由某个用户确认 |
| `rejected_by` | 动作由某个用户拒绝 |

实现建议：

- MVP 不必直接上图数据库。
- 先用 Postgres 表 + relation tables 建模。
- 把 graph 当成领域模型和索引层，而不是新的权威数据源。
- 飞书仍然是 OKR、任务、文档、会议的权威数据源。
- OAR Graph 是从飞书和用户反馈中派生出来的运营索引。

### 3.2 语义记忆 / Vector Memory

语义记忆用于从非结构化内容中找相关上下文。

适合进入向量层的内容：

- 文档摘要。
- 会议纪要摘要。
- 任务评论摘要。
- OKR progress 摘要。
- 历史 weekly review 片段。
- 用户对建议动作的拒绝原因摘要。

不建议默认进入向量层的内容：

- 完整会议原文。
- 完整 IM 原文。
- access token / refresh token。
- 敏感人事结论。
- 无权限跨团队内容。

推荐字段：

| 字段 | 说明 |
| --- | --- |
| `memory_id` | 语义记忆 ID |
| `tenant_id` | 租户隔离 |
| `source_type` | doc / meeting / task / okr_progress / decision_feedback |
| `source_ref` | 飞书对象引用 |
| `source_url` | 可回跳来源 |
| `summary` | 摘要文本 |
| `embedding` | 向量 |
| `linked_entity_ids` | 关联的 OKR / KR / User / Team |
| `visibility_scope` | 可见范围 |
| `created_at` | 创建时间 |
| `expires_at` | 过期时间，可选 |
| `retention_policy` | 保留策略 |

关键原则：

- 优先存摘要、来源引用和 hash，不默认保存完整原文。
- 检索结果必须再经过权限过滤。
- 向量相似不等于事实成立，最终回答必须回到证据链。

### 3.3 决策记忆 / Decision Memory

决策记忆记录人的判断和校准。

核心事件：

| 事件 | 说明 |
| --- | --- |
| `action_confirmed` | 用户确认了建议动作 |
| `action_edited_then_confirmed` | 用户编辑后确认 |
| `action_rejected` | 用户拒绝建议 |
| `evidence_trusted` | 用户认为某证据可信 |
| `evidence_dismissed` | 用户认为某证据不可信 |
| `preference_set` | 用户明确设置偏好 |
| `policy_override_requested` | 用户尝试越过策略，需要记录和拦截 |

推荐字段：

| 字段 | 说明 |
| --- | --- |
| `feedback_id` | 反馈 ID |
| `tenant_id` | 租户 |
| `user_id` | 做出判断的人 |
| `team_id` | 影响的团队 |
| `target_entity_id` | 相关 Objective / KR / Task / Doc |
| `proposed_action_id` | 原建议动作 |
| `decision` | confirmed / edited_then_confirmed / rejected |
| `reason_code` | too_aggressive / wrong_owner / weak_evidence / duplicate / not_important |
| `freeform_reason` | 用户自然语言原因 |
| `evidence_ids` | 用户看到的证据 |
| `created_at` | 发生时间 |

决策记忆可以学习：

- 用户偏好简短还是详细。
- 用户更愿意先评论、先建任务，还是先拉会。
- 哪些证据源可信。
- 哪类建议经常被拒绝。
- 某团队的运营节奏。

但要避免：

- 把一次拒绝过度泛化成长期规则。
- 把个人偏好错误应用到整个团队。
- 把敏感判断写成不可解释的隐藏画像。

## 4. 写入与更新流程

记忆不是一次性写入的静态知识库，而是随飞书数据同步、用户确认和审计事件持续更新的运营索引。

建议写入流程：

1. 数据进入：
   - 从 LarkAdapter 获取 OKR、任务、文档、会议、进展记录。
   - 从用户交互获取确认、编辑、拒绝和偏好反馈。

2. 权限过滤：
   - 所有数据先绑定 `tenant_id`、`user_id`、`visibility_scope`。
   - 无权限内容不进入检索索引。

3. 结构化归一：
   - 把飞书对象映射成 OAR 领域实体。
   - 建立 `owns`、`mentions`、`supports`、`derived_from` 等关系。

4. 摘要与向量化：
   - 对文档、会议、任务评论做摘要。
   - 保存摘要、来源引用、hash 和 embedding。

5. 决策反馈沉淀：
   - 用户确认、编辑、拒绝的动作写入决策记忆。
   - 拒绝原因和证据可信度反馈进入后续排序和生成策略。

6. 审计关联：
   - 所有影响写回的记忆引用都必须能关联到审计事件。
   - 记忆可重建，审计不可静默修改。

关键原则：

> 记忆层可以重建，审计层必须保真；记忆层用于提升判断质量，不能替代证据链和人的确认。

## 5. 检索与生成流程

一次 OKR 风险分析建议按这个顺序：

1. 从结构记忆找到相关实体和关系：
   - KR 属于哪个 Objective。
   - owner 是谁。
   - 关联任务、会议、文档有哪些。
   - 历史上有哪些建议动作。

2. 从语义记忆检索相关上下文：
   - 最近会议纪要中是否提到阻塞。
   - 任务评论里是否有进展。
   - 文档里是否解释了延期原因。

3. 从决策记忆读取人的偏好和历史判断：
   - 这个用户是否拒绝过类似提醒。
   - 这个团队更偏好哪种行动方式。
   - 哪些证据来源过去被认为不可信。

4. 生成风险判断和建议动作：
   - 给出风险等级。
   - 给出证据链。
   - 给出建议动作。
   - 给出置信度和不确定性。

5. 人确认、编辑或拒绝：
   - 确认后写回。
   - 编辑后确认进入新版本记录。
   - 拒绝原因进入决策记忆。

## 6. 与证据链的关系

记忆不是证据的替代品。

OAR 应该区分：

| 概念 | 作用 |
| --- | --- |
| 证据 | 支撑某次判断的具体来源 |
| 记忆 | 帮助系统更快找到证据、理解关系、学习人的偏好 |
| 审计 | 记录谁基于什么证据确认了什么动作 |

原则：

> 记忆可以影响检索和排序，但不能单独作为写回飞书的依据。

任何写回动作都必须能追溯到：

- 当前用户。
- 执行身份。
- scope。
- 建议动作。
- 证据链。
- 用户确认。
- 执行结果。

## 7. MVP 范围

MVP 不需要一次性做完整记忆系统。

推荐分阶段：

| 阶段 | 目标 | 记忆能力 |
| --- | --- | --- |
| 阶段 0.7 | 验证记忆是否提升建议质量 | Postgres 关系表 + pgvector + 决策反馈表 |
| 阶段 1 | 跑通真实每周复盘 | OKR / KR / owner / evidence / proposed action 基础关系 |
| 阶段 2 | 提升建议质量 | 加入用户拒绝原因、团队偏好、证据可信度 |
| 阶段 3 | A2A 只读开放 | 外部智能体只能读脱敏摘要，不读 raw memory |
| 阶段 4 | 多智能体协作 | 允许外部智能体提交建议，但仍由 OAR 决策记忆和人确认约束 |

MVP 最小集合：

- `Objective`
- `KeyResult`
- `User`
- `Evidence`
- `ProposedAction`
- `ConfirmedAction`
- `DecisionFeedback`

MVP 最小关系：

- `owns`
- `mentions`
- `supports`
- `derived_from`
- `confirmed_by`
- `rejected_by`

MVP 最小记忆能力：

- 找到某个 KR 相关的证据摘要。
- 找到某个 owner / team 过去被确认或拒绝的建议。
- 避免重复提出用户明确拒绝过的同类建议。

## 8. 实现建议

第一版推荐技术组合：

| 能力 | 建议实现 |
| --- | --- |
| 结构记忆 | Postgres 实体表 + relation tables |
| 语义记忆 | pgvector 或独立向量库 |
| 决策记忆 | append-only feedback/event 表 |
| 审计 | append-only audit event 表 |
| 权限过滤 | 所有查询强制带 `tenant_id`、`user_id`、`visibility_scope` |
| 删除和保留 | soft delete + retention job + source reindex |

暂不建议 MVP 直接上：

- 复杂图数据库。
- 自动抽取所有飞书消息成图谱。
- 跨租户全局记忆。
- 无法解释的长期个人画像。
- 让外部 A2A agent 直接访问记忆。

## 9. 风险与治理

| 风险 | 影响 | 治理方式 |
| --- | --- | --- |
| 记忆污染 | 错误建议越来越稳定 | 记录来源、置信度、人工反馈，支持回滚和重建 |
| 过度个性化 | 把个人偏好误当团队规则 | 偏好区分 user / team / tenant 层级 |
| 权限越界 | 跨团队泄露信息 | 检索前后都做权限过滤 |
| 向量召回幻觉 | 找到相似但无关内容 | 必须回到证据链，不用向量结果直接写回 |
| 敏感数据长期保留 | 企业合规风险 | retention policy、删除、导出、重建机制 |
| A2A 泄露记忆 | 外部智能体拿到内部运营偏好 | A2A 默认只返回脱敏摘要和 trace id |

## 10. 判断标准

记忆架构是否成立，不看它是否复杂，而看它是否让 OAR 在连续几周内变得更有用。

可观测指标：

- 重复错误建议减少。
- 用户拒绝原因被后续建议吸收。
- 同类 KR 风险判断更稳定。
- 证据召回更准确。
- 用户确认率提升。
- 人能解释系统为什么给出该建议。

最终判断：

> OAR 的长期壁垒不是保存更多数据，而是沉淀“这个组织如何运营目标执行”的可审计记忆。
