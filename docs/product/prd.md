# OAR MVP PRD

更新日期：2026-05-26
状态：Draft
阶段：阶段 0.6 身份与同步验证中

## 1. 摘要

OAR 是面向飞书企业租户的 OKR 复盘驾驶舱。MVP 聚焦一个高频场景：用户每周打开复盘收件箱，10 分钟处理完 OKR 执行风险和待确认动作。

第一版不替代飞书 OKR，不做 OKR 创建器，不做绩效评价。OAR 只做飞书之上的智能体工作流层：读取 OKR 与证据、识别风险、生成建议动作、经用户确认后安全写回飞书，并保留审计链路。

## 2. 背景与问题

很多团队已经在飞书中维护 OKR，但每周运营目标执行仍依赖人工：

- KR 长期没人更新，风险暴露太晚。
- 进展、阻塞和证据分散在 OKR、文档、任务、会议、妙记、日历和 IM 中。
- manager / PMO 需要手动整理周报、催进展、约同步、写评论。
- 企业不接受黑盒自动执行，需要权限继承、证据链、人工确认和审计。

问题陈述：

> 重度使用飞书并已有 OKR 节奏的团队，在每周复盘目标执行时，需要人工跨多个飞书产品收集证据和推动动作，导致风险发现慢、复盘准备成本高、跟进行动缺少闭环。

## 3. 目标与非目标

### 3.1 产品目标

- 让 manager / PMO 在每周复盘前快速看到有风险的 OKR。
- 把分散在飞书中的证据汇总成可解释的风险判断。
- 把提醒、评论、任务、会议草稿等跟进行动变成可确认的队列。
- 所有写回动作都有用户确认和审计事件。
- 验证用户是否愿意每周持续打开复盘收件箱。

### 3.2 非目标

- 不做通用 OKR SaaS。
- 不替代飞书 OKR。
- 不做 OKR 创建器。
- 不做绩效评价或个人绩效判断。
- 不做自动批量写回。
- 不开放外部 A2A 直接写回飞书。
- 不做复杂组织树和大型管理驾驶舱。

## 4. 目标用户

| 用户 | 核心场景 | 主要痛点 | MVP 价值 |
| --- | --- | --- | --- |
| 创始人 / CEO | 周会前看公司目标风险 | 信息分散，风险暴露晚 | 快速看到本周最高风险 OKR |
| 经理 / 团队负责人 | 跟进团队 OKR | owner 不更新，KR 进展缺证据 | 自动生成风险解释和跟进行动 |
| PMO / 幕僚长 | 运营组织 OKR 节奏 | 周报、催更、复盘靠人工 | 把 OKR 运营变成收件箱和审计流 |

优先 ICP：

> 20-300 人、重度使用飞书、已有 OKR 节奏但执行运营靠人工的团队。

## 5. 用户旅程

### 5.1 当前旅程

1. 用户准备周会或 OKR 复盘。
2. 用户打开飞书 OKR 查看 Objective / KR。
3. 用户切到任务、文档、会议、妙记、日历和 IM 查证据。
4. 用户手动判断哪些 KR 有风险。
5. 用户手动写评论、催 owner、建任务或约会议。
6. 跟进动作分散，后续难追溯谁基于什么证据做了什么。

### 5.2 MVP 旅程

1. OAR 后端按计划同步 OKR、任务、会议、文档和进展。
2. OAR 识别长期未更新、低进度、缺少 owner 更新的 KR。
3. OAR 生成风险队列、证据链和建议动作。
4. 用户打开复盘收件箱。
5. 用户查看证据，选择 `确认`、`编辑后确认` 或 `拒绝`。
6. 已确认动作通过 `LarkAdapter` 写回飞书。
7. OAR 记录 `AuditEvent` 并展示审计时间线。

## 6. MVP 范围

### 6.1 P0 范围

| ID | 需求 | 验收标准 |
| --- | --- | --- |
| P0-01 | 读取飞书 OKR 周期、Objective、KR 和 progress | 能基于 `lark-okr` 获取测试周期和详情 |
| P0-02 | 识别长期未更新 KR | 风险队列能展示至少一种 stale KR 风险 |
| P0-03 | 识别低进度 KR | 风险队列能展示低于预期节奏的 KR |
| P0-04 | 展示证据链 | 每条风险至少包含来源、摘要和回跳引用 |
| P0-05 | 生成建议动作 | 每条风险可生成评论、提醒、任务或会议草稿之一 |
| P0-06 | 支持确认、编辑后确认、拒绝 | 每条建议都有三种处理入口 |
| P0-07 | 写回前 dry-run | 写回前展示目标对象、payload 摘要和影响范围 |
| P0-08 | 用户确认后写回 progress 创建 / 更新 | 仅对确认动作执行，且可读回结果 |
| P0-09 | 幂等执行 | 同一 `ConfirmedAction` 并发确认只能产生一次执行记录 |
| P0-10 | 审计记录 | 每次写回记录 actor、scope、target、before/after 和执行结果 |
| P0-11 | 权限与 token 安全 | 客户端不保存飞书长期 token，日志不输出 token |

### 6.2 P1 范围

| ID | 需求 | 验收标准 |
| --- | --- | --- |
| P1-01 | 每周简报 | 生成团队级 OKR 风险摘要和本周建议动作 |
| P1-02 | owner 缺少更新检测 | 能识别 owner 长期无 check-in 或无 progress |
| P1-03 | 拒绝原因记录 | 用户拒绝建议时可记录原因并进入决策记忆 |
| P1-04 | 飞书消息卡片确认 | 用户能在飞书入口处理待确认动作 |
| P1-05 | 多端状态同步 | macOS、iOS、飞书入口看到一致动作状态 |

### 6.3 P2 范围

| ID | 需求 | 验收标准 |
| --- | --- | --- |
| P2-01 | 团队偏好学习 | 系统能减少重复提出已被拒绝的同类建议 |
| P2-02 | 只读 A2A 输出 | 外部智能体只能读取脱敏 OKR 摘要 |
| P2-03 | 高级证据可信度 | 用户可标记证据可信或不可信 |

## 7. 功能需求

### 7.1 复盘收件箱

- 系统必须按团队、周期和风险类型展示风险队列。
- 每条风险必须展示 Objective / KR、owner、风险原因、证据摘要和建议动作。
- 默认排序应优先展示影响大、长期未更新、低进度且证据充分的风险。
- 用户必须能按 `待确认`、`已确认`、`已拒绝`、`已执行`、`执行失败` 过滤。

### 7.2 证据链

- 每条风险判断必须绑定证据。
- 证据默认保存摘要、来源引用、hash 和可见范围，不默认保存完整原文。
- 证据来源至少覆盖 OKR progress、任务、会议纪要、文档摘要。
- 如果证据不足，系统必须明确展示不确定性，不能伪装成确定结论。

### 7.3 建议动作

- 系统可以自动生成建议动作，但不能自动写回。
- 建议动作类型包括：更新 progress、写 OKR 评论、提醒 owner、创建任务、创建会议草稿。
- MVP 默认只允许 progress 创建 / 更新进入真实写回。
- 删除 progress 只允许 dry-run 预览，不进入默认执行能力。

### 7.4 确认与编辑

- 每条建议必须支持 `确认`、`编辑后确认`、`拒绝`。
- 编辑后确认必须产生新的 action 版本，保留原建议。
- 拒绝建议必须允许记录原因。
- 确认动作必须生成 `ConfirmedAction` 和唯一 `idempotency_key`。

### 7.5 写回与审计

- 所有飞书调用必须经过 `LarkAdapter`。
- 写回前必须有 dry-run 结果。
- 后端必须通过 `OperationLedger` 保证同一动作只执行一次。
- 所有执行结果必须写入 append-only `AuditEvent`。
- 审计时间线必须能追溯：用户、执行身份、scope、目标对象、证据、动作、结果。

### 7.6 身份与授权

- OAR 必须区分 `user_delegated`、`bot_actor`、`app_actor` 和 `service_actor`。
- Lark 登录只绑定身份，不代表智能体自动拥有用户全部权限。
- 可用权限必须是 Lark app scopes、用户授权、资源权限和 OAR allowlist 的交集。
- 服务端保存 `TokenGrant`，客户端只保存 OAR session。
- refresh token rotation 必须原子保存新 token。

## 8. 非功能需求

| 类别 | 需求 | 目标 |
| --- | --- | --- |
| 安全 | 先读后写、写前预演、执行前人工确认 | 100% 写回来自 `ConfirmedAction` |
| 幂等 | 同一确认动作只执行一次 | 并发确认不产生重复写回 |
| 审计 | 写回全链路可追溯 | 100% 写操作有 `AuditEvent` |
| 权限 | 继承飞书权限并叠加 OAR 策略 | 无越权读取和写回 |
| 隐私 | 默认保存摘要和引用，不保存完整敏感原文 | 日志无 token 和敏感原文 |
| 可恢复 | CLI 输出变化可通过 fixture 回归发现 | parser 有样本测试 |
| 多端一致性 | 多端共享后端状态机 | 同一动作状态一致 |

## 9. 技术与数据要求

### 9.1 核心对象

- `Tenant`
- `OarUser`
- `LarkIdentity`
- `TokenGrant`
- `DeviceSession`
- `ProposedAction`
- `ConfirmedAction`
- `OperationLedger`
- `AuditEvent`
- `Evidence`
- `DecisionFeedback`

### 9.2 集成要求

- OKR 主路径使用 `lark-okr`，保留 OpenAPI 兜底。
- 当前已验证 progress 创建 / 更新可作为低风险写回路径。
- Lark CLI 必须锁定版本、保存 schema / output fixture，并做 parser 回归测试。
- `auth status`、dry-run 前缀、`progress_list` 字段、`content` 字符串二次解析都必须在 parser 中兼容。

### 9.3 记忆要求

OAR 记忆采用三层模型：

- 结构记忆：实体与关系。
- 语义记忆：摘要和证据检索。
- 决策记忆：确认、编辑、拒绝、偏好和信任校准。

记忆不能替代证据。任何写回动作仍必须追溯到当前证据链和用户确认。

## 10. 成功指标

MVP 值得继续推进的信号：

- 1 个真实团队连续 2-4 周使用。
- 复盘准备时间减少 50%。
- 建议动作确认或编辑后确认比例达到 30%+。
- 用户能解释为什么相信或不相信证据链。
- 每周至少 1 次主动复盘会话。
- 100% 写操作有确认记录和审计事件。

停止或回炉信号：

- 用户不愿每周打开复盘收件箱。
- 建议确认率长期低于 10%。
- 证据链无法让用户建立信任。
- 权限或 token 安全无法达到后台运行要求。
- 企业更愿意完全在飞书内完成流程，独立驾驶舱没有效率增益。

## 11. 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 飞书权限不足 | 无法形成证据链 | 先用最小 scopes 验证核心 OKR + progress 场景 |
| refresh token 无法安全保存 | 后台 7x24 不成立 | 阶段 0.6 优先实现 `TokenGrant` 加密存储和 revoke / reauth |
| 幂等执行不可靠 | 重复写回破坏信任 | 先实现 `OperationLedger` 并发测试，再开放多端确认 |
| 证据质量不足 | 建议不可信 | 明确展示不确定性，优先摘要和来源引用 |
| 用户不形成每周习惯 | 产品切口不成立 | 陪跑 3-5 位真实用户，验证连续使用 |
| 产品范围膨胀 | 过早进入复杂平台 | MVP 禁止 A2A 写回、绩效、复杂组织仪表盘 |
