# OAR 一页版项目简报

更新日期：2026-05-26

## 1. 一句话

OAR 是面向飞书企业租户的 **OKR 复盘驾驶舱**：每周自动发现 OKR 执行风险，汇总证据，生成行动建议，并在用户确认后安全写回飞书。

## 2. 要解决的问题

很多团队已经在飞书里写了 OKR，但真正痛苦的不是创建 OKR，而是每周持续运营它：

- KR 长期没人更新，风险暴露太晚。
- 进展、阻塞和证据分散在 OKR、文档、任务、会议、妙记、日历和 IM 里。
- manager / PMO 需要人工整理周报、催进展、约同步、写评论。
- AI 能生成总结，但企业需要权限继承、证据链、人工确认和审计，而不是黑盒自动执行。

OAR 的核心假设是：

> OKR 的主要机会不在“帮人写目标”，而在“帮人每周运营目标执行风险”。

## 3. 目标用户

优先服务 20-300 人、重度使用飞书、已有 OKR 节奏但执行运营靠人工的团队。

第一批用户画像：

- 创始人 / CEO：想在周会前看到公司目标风险。
- 经理 / 团队负责人：想快速知道哪些 KR 卡住、谁需要跟进。
- PMO / 幕僚长：想把 OKR 复盘、催更和行动闭环从人工表格变成收件箱。

## 4. 产品形态

第一版只做一个高频场景：

> 每周打开 OAR，10 分钟处理完所有 OKR 风险和待确认动作。

默认入口不是聊天框，也不是指标大屏，而是 **复盘收件箱**。

用户每周看到：

- 本周 OKR 风险队列。
- 每条风险背后的证据链。
- 智能体生成的建议动作。
- 待确认的写回动作。
- 审计时间线。

每条建议都必须支持 `确认`、`编辑后确认`、`拒绝`。所有真实写回都必须经过人工确认。

## 5. 为什么不是飞书 OKR 替代品

OAR 不替代飞书 OKR。飞书仍然是 OKR、文档、任务、会议、日历和 IM 的权威数据源。

OAR 只做飞书之上的智能体工作流层：

- 读取飞书中的目标和证据。
- 发现执行风险。
- 起草行动建议。
- 通过受控工具层写回飞书。
- 保留确认记录和审计链路。

这让 OAR 避开通用 OKR SaaS 和绩效系统的正面战场，聚焦飞书租户内的执行运营闭环。

## 6. MVP 包含什么

MVP 必须包含：

- 从飞书读取 OKR 周期、Objective、KR 和 progress。
- 识别长期未更新的 KR、低进度 KR、缺少更新的 owner。
- 聚合 Docs、Tasks、Meetings、Minutes、Calendar、IM 中的相关证据摘要。
- 生成每周简报、风险队列和建议动作。
- 用户确认后写回进度、评论、提醒、任务或会议草稿。
- 记录 `AuditEvent`，追溯 actor、scope、target、before/after 和执行结果。

MVP 不做：

- OKR 创建器。
- 绩效评价。
- 自动批量写回。
- 通用智能体市场。
- 外部 A2A 写回飞书。

## 7. 技术判断

推荐技术形态：

- macOS：SwiftUI + AppKit bridge。
- 后端 / Core：Rust service。
- 飞书集成：`LarkAdapter` 封装 Lark CLI 和 OpenAPI。
- 存储：Postgres + 对象存储 + 向量索引。
- 智能体运行时：服务端 7x24 调度、同步、审计和工具执行。

关键原则：

- 业务代码不直接调用 CLI 或 OpenAPI，必须经过 `LarkAdapter`。
- Lark CLI 可以作为智能体工具层，但生产必须锁定版本、保存 schema 快照，并保留 OpenAPI 兜底路径。
- 所有写回必须来自 `ConfirmedAction`。
- 客户端只负责体验、查看和审批，不能承担组织级后台运行。

## 8. 当前最大风险

OAR 当前处于阶段 0.6：身份与同步验证。阶段 0.5 已确认 `lark-okr` 可以作为 OKR 只读主路径和 progress 创建 / 更新写回路径。

当前最大不确定性已经从 OKR CLI 能力转向生产级身份、安全执行和用户习惯：

- `TokenGrant` 能否安全保存、刷新和撤销。
- `OperationLedger` 能否保证同一确认动作只执行一次。
- macOS、iOS 和飞书入口能否看到一致的动作状态。
- 后台 worker 能否在客户端离线时可靠运行。
- 用户是否愿意每周打开复盘收件箱并处理建议动作。

## 9. 接下来 7 天

1. 固化阶段 0.5 的 CLI 输出，作为 `LarkAdapter` parser 和 fixture 的回归样本。
2. 设计 `TokenGrant`、`DeviceSession`、`OperationLedger` 和 `AuditEvent` 的最小 schema。
3. 实现 `ConfirmedAction -> OperationLedger -> LarkAdapter -> AuditEvent` 的最小状态机。
4. 用并发测试验证两个入口同时确认同一动作时只写回一次。
5. 用假数据搭建复盘收件箱原型，验证用户是否理解“风险队列 + 证据链 + 待确认动作”。

## 10. 成功信号

如果 1 个真实团队连续 2-4 周使用，并出现以下信号，OAR 值得继续推进：

- 复盘准备时间减少 50%。
- 建议动作确认或编辑后确认比例达到 30%+。
- 用户能解释为什么相信或不相信证据链。
- 每周至少 1 次主动复盘会话。
- 100% 写操作有确认记录和审计事件。

如果用户不愿每周打开复盘收件箱，或者只读简报无法形成稳定习惯，应暂停扩大技术范围，重新验证产品切口。
