# OAR 产品计划

更新日期：2026-05-26  
当前状态：方向判断 + MVP 定义 + Phase 0.5 验证准备  
配套执行清单：[`docs/lark-capability-matrix.md`](docs/lark-capability-matrix.md)

## 0. 核心结论

OAR 值得继续做，但它不应该是通用 OKR SaaS，也不应该是通用 Agent Desktop。

最清晰的产品切口是：

> 面向飞书企业租户的 **OKR Review Cockpit**：每周自动发现 OKR 执行风险，汇总证据，生成行动建议，并在用户确认后安全写回飞书。

长期形态是：

> 飞书 OKR 的 **AI Chief of Staff + Agent Gateway**：负责盯进展、找风险、起草动作，并让其他 agents 在 A2A 边界内协作；关键写回永远由人确认。

一句话原则：

> 不是让用户“管理 OKR”，而是让用户每周 10 分钟清空 OKR 执行风险队列。

当前最重要的下一步：

> 完成 Phase 0.5，用真实飞书租户验证 `lark-okr` 的 read/write/scope/dry-run/schema 能力。未验证前，所有 Lark CLI 能力都只能视为设计假设。

## 1. 产品决策

| 主题 | 决策 | 原因 |
| --- | --- | --- |
| 产品定位 | OKR Review Cockpit | 避开通用 OKR SaaS 和通用 Agent Desktop，聚焦飞书内的 OKR 执行运营 |
| System of record | 飞书 OKR | OAR 不替代飞书，只做 Agent operating layer |
| MVP 场景 | Weekly OKR Review | 高频、可验证、能体现 Agent 价值 |
| 默认入口 | Review Inbox | 避免变成低频 dashboard |
| 写回策略 | read-first, dry-run-before-write, human-confirm-before-execute | 建立企业信任和审计边界 |
| 技术主线 | Swift 原生前端 + Rust backend/core + LarkAdapter | 保持原生体验、强类型工具层和可审计执行 |
| Lark 集成 | LarkAdapter-first；OKR 路径优先验证 Lark CLI，保留 OpenAPI fallback | Lark CLI 适合 Agent Tool Layer，但生产必须可替换 |
| 7x24 运行 | 后端 Agent Runtime，而不是桌面端常驻 | 客户端负责交互和审批，后台负责调度、同步、审计 |
| A2A | Phase 3 以后只读开放 | MVP 先闭环飞书 OKR，避免过早扩大安全面 |

## 2. 已验证与待验证假设

| 假设 | 状态 | 当前判断 |
| --- | --- | --- |
| 飞书 OKR 仍是主要 system of record | 公开资料支持 | 成立，OAR 应叠加在飞书之上 |
| Lark CLI 可作为 Agent Tool Layer | 公开资料支持 | 方向成立，但必须 pin version、做 schema snapshot 和 regression |
| `lark-okr` 覆盖 OKR 周期、Objective、KR、进展记录 | 部分公开资料支持 | 必须用真实租户 smoke test 验证 |
| Lark OAuth 可绑定用户身份 | 公开文档支持 | 可以做 user-delegated identity |
| 后台 7x24 可代表用户工作 | 有条件成立 | 依赖 `offline_access`、refresh token rotation 和企业合规授权 |
| 飞书事件订阅可支撑 Reactive Agent | 未充分验证 | MVP 采用 scheduled-first，事件订阅作为增强 |
| 用户愿意每周打开 Review Inbox | 未验证 | 需要 3-5 个真实 manager / PMO 做 2-4 周 concierge MVP |
| 企业允许保存 token、evidence summary、memory | 未验证 | 先做 single-tenant / internal pilot，并设计删除、导出、retention |
| Memory 能显著提升建议质量 | 未验证 | 需要 historical OKR review regression cases |

## 3. 市场判断

已看到的相似方向：

| 产品/类型 | 相似点 | OAR 应避开的正面战场 |
| --- | --- | --- |
| WorkBoardAI | AI strategy execution、pre-read、scorecards | 企业战略执行大平台 |
| Tability / Tabby AI | OKR Agent、check-in、行动建议、MCP | 通用 OKR Agent |
| Rhythms.ai | AI-native OKR、nudges、自动进度 | AI OKR 平台 |
| Betterworks / Leapsome / Lattice | OKR + performance management | HR / 绩效管理 |
| 飞书 OKR | OKR system of record | OKR CRUD 和组织权限 |
| OpenClaw | 桌面 agent cockpit、skills、tool 管理 | 通用 agent workspace |

结论：

- AI 生成 OKR、自动报告、check-in reminder 已不稀缺。
- OAR 的差异点必须是飞书 tenant 内的深集成、权限继承、证据链、确认写回和审计。
- 不碰绩效评价是必要边界，否则会进入高敏 HR 场景，降低试点速度。
- 桌面端可以像 OpenClaw 一样是 cockpit，但 OAR 必须保持 OKR 垂直，不做通用电脑控制。

## 4. 目标用户

| 用户 | 核心场景 | 痛点 | OAR 价值 |
| --- | --- | --- | --- |
| Founder / CEO | 每周看公司目标风险 | 信息散在飞书各处，风险暴露晚 | 看到本周 at-risk OKR 和建议动作 |
| Manager / Team Lead | 跟进团队 OKR | owner 不更新，KR 进展缺证据 | 自动生成 review 和 follow-up action |
| PMO / Chief of Staff | 运营组织 OKR 节奏 | 周报、催更、复盘靠人工 | 把 OKR 运营变成 inbox 和 audit flow |
| AI Agent / 外部系统 | 获取 OKR 上下文或提交建议 | 不知道如何安全读写 OKR | 通过 A2A 请求受控 OKR skills |

优先 ICP：

> 20-300 人、重度使用飞书、已有 OKR 节奏但执行运营靠人工的团队。第一批最好是内部团队或 design partner，不建议直接做开放 SaaS。

## 5. MVP 定义

第一版只验证一个场景：

> 每周打开 OAR，10 分钟处理完所有 OKR 风险和待确认动作。

MVP 必须包含：

- 从飞书读取 OKR 周期、Objective、KR 和 progress。
- 识别 stale KR、低进度 KR、缺少更新的 owner。
- 聚合 Docs、Tasks、Meetings、Minutes、Calendar、IM 中的相关证据摘要。
- 生成 weekly brief、risk queue 和 proposed action。
- 用户确认后写回进度、评论、提醒、任务或会议草稿。
- 记录 `AuditEvent`，可追溯 actor、scope、target、before/after 和执行结果。

MVP 不做：

- OKR 创建器。
- 完整组织树和复杂 dashboard。
- 绩效评价。
- 自动批量写回。
- 通用 Agent marketplace。
- 外部 A2A 写回飞书。

第一版交付物：

- macOS Review Cockpit。
- Rust backend/core 的 `LarkAdapter`、Agent Runtime、Audit、Queue。
- 飞书 OAuth 登录和 identity binding。
- Phase 0.5 Lark CLI 能力验证报告。
- 1 个真实团队跑通 2-4 周 weekly loop。

## 6. 桌面端形态

桌面端应该是 **OKR Review Cockpit**，不是报表首页，也不是聊天首页。

推荐三栏布局：

| 区域 | 作用 |
| --- | --- |
| 左侧 Sidebar | 团队、周期、风险视图、Pending Actions、Audit、Agent Network |
| 中间主区 | OKR review board、风险排序、目标详情、KR 进度、证据链 |
| 右侧 Agent Panel | Agent 解释、建议、proposed action、confirm/reject、执行日志 |

默认入口：

- Review Inbox。
- 本周 OKR 风险榜。
- Pending Agent Actions。
- Weekly Brief。

点进单个 OKR 后，用户应该看到：

- Objective / KR 当前状态。
- Agent 判断：为什么 at risk。
- Evidence：更新时间、任务进度、会议纪要、文档变更、owner check-in。
- Agent 建议：更新进度、写评论、提醒 owner、创建 follow-up task。
- 操作：`Confirm`、`Edit then Confirm`、`Reject`、`Ask Agent`。

桌面端不是“万能聊天框”。聊天只能作为 explain / refine / ask follow-up 的辅助能力，主工作流必须是 inbox、queue、timeline 和 confirm。

## 7. iOS 与飞书入口

iOS 只做轻量 companion：

- 查看今日风险。
- 接收提醒。
- approve/reject pending action。
- 快速查看 weekly brief。

飞书入口是高优先级：

- Bot 做触达。
- 消息卡片做确认。
- Shortcut 做快速 review。
- 重要动作回到 OAR 或飞书卡片确认。

产品心智：

> OAR Agent 在后台持续准备工作；用户只在需要判断和授权时出现。

## 8. 原生技术形态

推荐实现：

| 层 | 技术 | 说明 |
| --- | --- | --- |
| macOS | SwiftUI + AppKit bridge | 原生窗口、sidebar、command menu、通知、deep link |
| iOS | SwiftUI | 轻量审批和提醒，不承载完整 Agent Runtime |
| Backend / Core | Rust | LarkAdapter、queue、audit、policy、tool execution、sync engine |
| 本地通信 | local HTTP / gRPC / XPC | macOS shell 可连本地 core；生产更推荐后端服务 |
| Server | Rust service | 7x24 jobs、OAuth token、sync、A2A server、audit store |
| Storage | Postgres + object store + vector index | audit、pending action、evidence summary、memory |

关键判断：

- iOS 不适合作为 7x24 Agent 执行环境，只作为审批和查看端。
- macOS 可以有 local core，但不能依赖用户电脑在线完成组织级 review。
- 真实 7x24 能力必须在 server-side Agent Runtime。
- Swift 前端只负责体验和状态展示，所有写回都通过 backend 的 `ConfirmedAction`。

## 9. Lark CLI-first 原则

OAR 是基于 Lark CLI 开发的应用，但不能只是“调用 CLI”。业务层必须 **LarkAdapter-first**，OKR 主路径在 Phase 0.5 优先验证 Lark CLI，失败时切换或降级到 OpenAPI。

核心技术原则是：

> 把 Lark CLI 产品化成安全、可审计、可回滚的 Agent Tool Layer。

公开资料支持的方向：

- Lark CLI 覆盖 Messenger、Docs、Drive、Sheets、Base、Calendar、Meetings、Minutes、Mail、Tasks、Wiki、Contacts、Approval、Attendance、OKR 等核心域。
- 官方资料显示其支持大量 commands、Agent skills、dry-run、schema introspection、auth check、identity switching。
- `lark-okr` 是 OKR 主路径的优先验证对象。

保守原则：

- 当前能力以本机安装版本的 schema snapshot 为准。
- 生产环境 pin CLI version，不自动升级。
- 所有调用经过 `LarkAdapter`。
- 每个 command 都要有 allowlist、timeout、output limit、脱敏和 audit。
- OpenAPI fallback 必须设计进架构，不能把 CLI 当成不可替换黑盒。

Phase 0.5 详细验证见：[`docs/lark-capability-matrix.md`](docs/lark-capability-matrix.md)

## 10. LarkAdapter

所有飞书能力必须经过 `LarkAdapter`，业务代码不直接散落调用 CLI 或 OpenAPI。

| Adapter | 责任 |
| --- | --- |
| `OkrAdapter` | 读取周期、Objective、KR、progress、alignment；必要时写回低风险进度或评论 |
| `EvidenceAdapter` | 从 Docs、Tasks、Meetings、Minutes、IM、Calendar 收集证据 |
| `ActionAdapter` | 将 proposed action 转成任务、提醒、评论、会议草稿 |
| `AuthAdapter` | 管理 app/user/bot identity、scope 检查、缺权限引导 |
| `AuditAdapter` | 记录 CLI/OpenAPI 调用、dry-run、confirm、执行结果 |

最小接口：

```text
list_okr_cycles(user_id, user_id_type) -> OkrCycle[]
get_okr_cycle_detail(cycle_id) -> OkrCycleDetail
list_progress(target_id, target_type) -> ProgressRecord[]
dry_run_create_progress(request) -> ToolDryRun
create_progress(confirmed_action_id, request) -> ProgressRecord
dry_run_update_progress(request) -> ToolDryRun
update_progress(confirmed_action_id, request) -> ProgressRecord
```

所有写方法必须接收 `confirmed_action_id`，并写入 `AuditEvent`。

## 11. 账号、身份与 7x24

OAR 不应该依赖桌面端常驻来实现智能。更合理的形态是：

> macOS / iOS / 飞书入口负责交互、查看和审批；后端 Agent Runtime 负责 7x24 调度、同步、审计和工具执行。

账号模型：

| 对象 | 作用 |
| --- | --- |
| `Tenant` | 飞书企业租户，数据隔离和计费边界 |
| `OarUser` | OAR 内部用户，保存偏好、角色、memory 设置 |
| `LarkIdentity` | 飞书用户身份绑定，如 `open_id`、`union_id`、tenant 信息 |
| `RoleBinding` | manager、PMO、admin、viewer 等 OAR 内角色 |
| `DeviceSession` | macOS / iOS / web / 飞书入口的登录设备和会话 |
| `TokenGrant` | 用户或企业授权给 OAR 的 OAuth grant，必须加密存储 |
| `AgentActor` | Agent 执行时使用的身份描述，不直接等同于某个 token |

执行身份：

| 身份 | 来源 | 适合场景 |
| --- | --- | --- |
| `user_delegated` | 用户 Lark OAuth 登录后授权 | 读取用户可见 OKR、日历、任务、文档；执行用户确认后的写回 |
| `bot_actor` | 飞书 Bot / 企业自建应用 | 发消息卡片、提醒、确认入口、系统通知 |
| `app_actor` | 企业应用授权 | 组织级同步、公开团队 OKR、后台批处理 |
| `service_actor` | OAR 后端内部身份 | schedule、queue、audit、model job |
| `approved_user_action` | 用户确认后的 action | 写评论、建任务、提醒 owner、更新低风险字段 |

关键原则：

- Lark 登录可以自动绑定用户身份，但不能自动放大成“Agent 拥有该用户全部权限”。
- 实际可用权限必须取 Lark app scopes、用户授权、资源权限和 OAR policy allowlist 的交集。
- 如果需要 7x24 后台运行，需要申请 `offline_access`。
- `refresh_token` 只在用户授予 `offline_access` 时返回。
- 刷新成功后必须保存新的 refresh token，原 refresh token 可能失效。
- 客户端只保存 OAR session，不长期保存飞书 refresh token。

## 12. 多端同步

飞书与 OAR 的职责分界：

| 系统 | 作为 system of record 的数据 |
| --- | --- |
| 飞书 | OKR、Docs、Tasks、Meetings、Calendar、IM 原始数据 |
| OAR backend | Weekly Review、Pending Actions、Audit Events、Evidence Index、Memory、sync cursor |
| 客户端 | UI cache、draft、device session，不保存长期 token |

同步原则：

- 所有客户端通过 `sync_cursor` 拉取增量状态。
- 写操作必须使用 `idempotency_key`。
- 后端维护 `OperationLedger`，记录 pending、confirmed、executing、succeeded、failed、cancelled。
- 同一个 pending action 在 macOS / iOS / 飞书卡片中的状态必须一致。
- 客户端离线时只能编辑 draft，不能离线确认真实写回。

## 13. Agent Runtime、模型配置与 Memory

OAR 的智能不应该来自一次 LLM 调用，而应该来自：

> 模型编排 + 可追溯证据 + 可学习的团队记忆 + 可版本化 policy。

模型角色：

| 模型角色 | 用途 | 要求 |
| --- | --- | --- |
| `fast_model` | 分类、摘要、stale 检测、低风险初筛 | 低延迟、低成本、稳定 JSON |
| `reasoning_model` | weekly review、复杂风险判断、proposed action | 推理强、上下文长、结构化输出 |
| `embedding_model` | evidence retrieval、memory retrieval | 向量质量稳定、可批量处理 |
| `local_or_private_model` | 高敏企业数据场景 | 可私有化、可审计、可按 tenant 关闭外部模型 |

每次生成 review 或 proposed action 时记录：

- `prompt_version`
- `policy_version`
- `model_provider`
- `model_version`
- `tool_schema_version`
- `output_schema_version`
- `evidence_ids`
- `memory_ids`
- `trace_id`

Memory 分层：

| Memory 类型 | 内容 | 用途 |
| --- | --- | --- |
| User Preference Memory | brief 风格、提醒偏好、常用确认方式 | 个性化输出 |
| Team Operating Memory | 团队节奏、owner 分工、例会规律 | 理解团队上下文 |
| OKR Cycle Memory | 本周期目标、历史 review、progress 变化 | 跨周风险判断 |
| Decision Memory | 用户确认/拒绝过的 action 和原因 | 避免重复建议 |
| Policy Memory | 企业禁止动作、审批规则、scope 约束 | 防止越权 |

Memory 必须支持关闭、删除、导出、重建、retention policy 和 tenant 隔离。MVP 可以先只做 `Decision Memory` 与 `OKR Cycle Memory`，避免过早做复杂个人画像。

## 14. Evidence Chain

OAR 的每条 Agent 判断都必须绑定来源。

| Agent 判断 | 可用证据来源 |
| --- | --- |
| KR stale | OKR 更新时间、owner 最近消息、任务更新时间 |
| KR 低于预期节奏 | OKR progress、任务完成率、会议/文档中的 blocker |
| owner 未响应 | IM 搜索、任务评论、OKR 评论、会议纪要 |
| 目标缺证据 | Docs/Wiki/Minutes/Tasks 中找不到关联上下文 |
| 需要 follow-up | Minutes todos、Calendar 空档、Tasks 未完成项 |
| 可以更新 progress | 已完成任务、会议结论、owner check-in、Base/Sheets 指标 |

Evidence Store 最小字段：

- `evidence_id`
- `tenant_id`
- `source_type`
- `source_ref`
- `source_url`
- `source_timestamp`
- `collected_at`
- `linked_okr_id`
- `linked_kr_id`
- `owner_user_id`
- `extracted_summary`
- `raw_excerpt_hash`
- `confidence`
- `visibility_scope`
- `retention_policy`

MVP 原则：优先保存摘要、来源引用和 hash，不默认保存完整原文。

## 15. Execution Safety Model

默认原则：

> read-first, dry-run-before-write, human-confirm-before-execute.

执行链路：

1. Agent 产生 intent，例如“提醒 owner 更新 KR”。
2. OAR 将 intent 转成受控 tool request。
3. `LarkAdapter` 生成 dry-run 结果。
4. OAR 将 dry-run 结果展示为 `ProposedAction`。
5. 用户 `Confirm` 或 `Edit then Confirm`。
6. Backend 执行 allowlist 中的 CLI/OpenAPI 操作。
7. 写入 `AuditEvent`，并显示到 audit timeline。

必须限制：

- CLI command allowlist。
- target object allowlist。
- scope allowlist。
- tenant/user context 校验。
- timeout、retry、rate limit。
- output size limit 和敏感信息脱敏。

禁止：

- LLM 直接执行任意 raw command。
- LLM 自动放宽 scope。
- 未经确认发送群消息、改 OKR、建会议、批量改任务。
- 外部 A2A agent 直接拿到 CLI stdout/stderr 或任何 identity token。

## 16. Agent 能力边界

| 层级 | Agent 能做什么 | 是否自动执行 |
| --- | --- | --- |
| L1 Observe | 读取 OKR、任务、会议、文档、评论、更新时间 | 自动 |
| L2 Diagnose | 判断风险、发现 stale KR、识别 blocker、生成 review | 自动 |
| L3 Propose | 生成行动建议、更新草稿、评论草稿、任务草稿 | 自动生成，不写回 |
| L4 Act | 写回 OKR、发评论、建任务、约会议、通知成员 | 必须用户确认 |
| L5 Autonomous Act | 预授权低风险动作 | MVP 不做 |

第一版可以做：

- 发现长期未更新的 OKR。
- 判断 KR 是否低于预期节奏。
- 根据任务、会议、文档生成 weekly check-in 草稿。
- 提醒 owner 补充进展。
- 建议更新 KR 进度。
- 建议创建 owner sync 会议。
- 建议在飞书 OKR 下写进展评论。

第一版不应该做：

- 自动创建或删除 Objective。
- 自动修改 KR target、权重、owner、周期。
- 自动评价个人绩效。
- 自动群发敏感结论。
- 自动跨部门读取无权限数据。
- 自动执行批量 OKR 变更。

## 17. A2A 策略

引入 A2A 后，OAR 的长期形态是 **OKR Agent Hub**。

协议分工：

- MCP / Lark CLI / OpenAPI 是工具层。
- A2A 是协作层。
- OAR 是权限、确认、审计和 OKR domain policy 的控制层。

OAR 对外可暴露的 A2A skills：

- `okr.risk_review`
- `okr.weekly_brief`
- `okr.progress_diagnosis`
- `okr.propose_action`
- `okr.audit_status`

外部 A2A agent：

- 只能通过授权 skill 读取摘要。
- 可以请求 OAR 生成 proposal。
- 永远不能直接写回飞书。
- 不能读取 raw evidence。
- 不能访问 OAR memory。
- 不能绕过 Pending Action。

路线：

- Phase 1-2：不开放外部 A2A，只做内部 Agent workflow。
- Phase 3：只读 A2A Server，输出 weekly brief、risk summary、audit status。
- Phase 4：允许外部 agent 提交 `ProposedAction`，但仍由 OAR 用户确认后执行。

## 18. 权限与数据边界

基本原则：

- OAR 代表当前登录用户或被授权的企业应用读取飞书数据。
- 每个 request 必须绑定 `tenant_id`、`user_id`、scope 和 actor。
- manager 只能看到自己有权限查看的团队和成员 OKR。
- 外部 A2A agent 不能直接读取原始 OKR 数据。
- 所有写回必须来自 `ConfirmedAction`。

数据分级：

| 数据 | 示例 | 处理规则 |
| --- | --- | --- |
| 原始 OKR | Objective、KR、owner、进度 | 仅 OAR backend 和授权用户可见 |
| 证据数据 | 文档摘要、会议纪要、任务状态 | 尽量摘要化，保留来源引用 |
| Agent 建议 | risk、rationale、proposed action | 展示给授权用户，可进入 pending queue |
| Audit 数据 | actor、before/after、时间、结果 | append-only，禁止静默修改 |
| A2A 输出 | weekly brief、risk summary、artifact | 默认脱敏和最小化 |

永不允许：

- 外部 agent 直接持有飞书长期 token。
- 外部 agent 绕过 OAR 写回飞书。
- 未经授权读取跨团队/跨部门 OKR。
- 在日志中输出 access token、refresh token、完整会议原文或敏感人事结论。

## 20. MVP 验证实验

第一阶段不要验证完整技术栈，而要验证用户是否真的需要 Weekly OKR Review Cockpit。

Concierge MVP：

1. 找 3-5 位真实目标用户：manager、PMO、Chief of Staff、Founder。
2. 选 1-2 个真实团队和一个 OKR 周期。
3. 手动或半自动读取飞书 OKR、任务、会议、文档更新。
4. 用 OAR prototype 生成 weekly brief、风险榜和 proposed actions。
5. 在飞书或 macOS prototype 中让用户确认、编辑或拒绝 action。
6. 记录用户是否愿意下周继续使用，以及哪些建议被采纳。

通过信号：

- review 准备时间减少 50%。
- proposed action confirm 或 edit-then-confirm 比例达到 30%+。
- 用户能解释为什么相信或不相信 evidence chain。
- 每周至少 1 次 active review session。
- 用户愿意让 OAR 继续进入下一个 OKR 周期。

## 21. 成功指标

| 目标 | 指标 |
| --- | --- |
| 节省 review 时间 | 每周 OKR review 准备时间减少 50% |
| 提高风险发现 | at-risk KR 在周会前被识别的比例提升 |
| 建立信任 | proposed action confirm rate 达到 30%-50% |
| 形成习惯 | 每周至少 1 次 active review session |
| 安全可控 | 100% 写操作有确认记录和 audit event |
| 飞书闭环 | confirmed action 成功写回飞书比例稳定 > 95% |
| 多端一致性 | 同一个 pending action 在 macOS / iOS / 飞书卡片中状态一致 |
| 常驻可靠性 | scheduled weekly review job 按时完成率 > 95% |
| 身份可信 | 100% 写操作能追溯到授权用户、执行身份、scope 和 confirmed action |
| 记忆有效性 | 被用户拒绝过的重复建议明显减少 |

## 22. 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 飞书 OKR API 写权限难申请 | 无法完整写回 | MVP 先支持只读洞察 + 评论/任务/提醒写回 |
| Lark CLI 能力与权限不稳定 | evidence chain 无法闭环 | Phase 0.5 单独验证 `lark-okr`、Docs、Tasks、Minutes、Calendar、IM |
| 未获得 `offline_access` | 无法后台 7x24 刷新用户凭证 | Phase 0.6 go/no-go；缺失时降级为用户在线触发 |
| OAuth 实际 scope 被裁剪 | Agent 以为有权限但执行失败 | 以 token 响应 `scope` 为准，建立 missing scope 引导和 fallback |
| 用户不信任 AI 判断 | confirm rate 低 | 每条建议必须给 evidence 和 before/after |
| 变成又一个 dashboard | 使用频率低 | 首页必须是 Review Inbox，不是指标大屏 |
| 组织数据敏感 | 企业不敢授权 | scope 最小化、tenant 隔离、审计、人工确认 |
| refresh token 泄露或滥用 | 极高安全风险 | 加密存储、最小 scope、rotation、revoke、日志脱敏、异常使用告警 |
| 多端重复确认 | 重复发消息或重复写回 | `OperationLedger`、`idempotency_key`、后端状态机 |
| Memory 泄露或跨租户污染 | 高敏数据风险 | tenant 隔离、retention policy、可删除/导出、A2A 默认不读 memory |
| A2A 过早复杂化 | 产品范围失控 | Phase 3 前只做内部 agent，不开放外部 A2A |
| 与飞书原生 OKR 重叠 | 价值不清 | 明确不做 OKR CRUD，只做执行风险和行动闭环 |

## 23. Kill Criteria

如果出现以下情况，应考虑停止或转向：

- 拿不到足够的飞书 OKR / 任务 / 文档读取权限，导致无法形成 evidence chain。
- `lark-okr` 与 OKR OpenAPI 都无法稳定拿到 OKR 主数据。
- 无法取得 `offline_access` 或无法合规保存/刷新用户授权，导致 7x24 backend Agent 不能成立。
- 用户每周不愿打开 Review Inbox，说明工作流切口不成立。
- proposed action 的确认率长期低于 10%，说明建议不可用或不可信。
- 多端同步无法避免重复执行或状态不一致。
- Memory 和 Evidence Store 无法让建议质量随周期提升。
- 写回权限长期无法落地，且只读 brief 不能形成稳定使用习惯。
- 企业更愿意直接在飞书内完成全部流程，独立桌面端没有明显效率增益。

## 24. 未来判断

**有未来，但它不是一个大众软件机会，而是一个垂直企业工作流机会。**

它的未来取决于三个判断是否成立：

1. OKR 的真正痛点不是创建，而是执行运营。
2. 飞书生态足够封闭且足够丰富。
3. Agent 的价值在“起草 + 证据 + 确认”，不是全自动。

明确判断：

> 值得继续做，但第一版必须极窄：只做 Weekly OKR Review Cockpit。先证明“每周 10 分钟清空 OKR 风险队列”这件事有人愿意持续使用，再谈 A2A 和更大的 Agent Hub。

如果 weekly loop 跑通，OAR 有机会从一个飞书内部工具长成 OKR Agent Hub；如果跑不通，继续加 Agent、A2A、原生端都只是把复杂度堆高。
